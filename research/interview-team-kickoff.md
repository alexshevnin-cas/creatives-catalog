**Статус:** Не решено. Alexei: _"Надо подумать."_

### R3. Медиафайл отвязать от кода 1С (реляционная связь)
Один и тот же видеофайл может использоваться для iOS и Android версий приложения, у которых **разные коды 1С**. Если код 1С зашит в имя файла — возникают дубликаты.

**Предложение Kapatsyn + Максим:** Медиафайл — отдельная сущность. Креатив ссылается на медиафайл через реляционную связь. Файл хранит только свои характеристики (разрешение, длительность, формат). Код 1С — атрибут креатива, не файла.

_"Медиафайл не должен быть жёстко завязан по айдишке. Он должен быть сам по себе."_ — Ruslan

**Статус:** Требует проработки. Alexei: _"Уточню процесс."_

### R4. Уникальный системный ID в имени файла
Идея: добавить короткий хэш/ID (~8 символов) в начало или конец имени файла. Система ориентируется по хэшу, а человекочитаемая часть — для менеджеров.

_"Вначале пишем уникальный айдишник нашей внутренней системы, а в конце уже лабуда — разрешение и что захотите."_ — Ruslan

**Статус:** Не решено. Идея зафиксирована, требует валидации с UA-менеджерами.

### R5. Загрузка по ссылке Google Drive
Вместо скачивания файла на локальный компьютер и повторной загрузки в каталог — поддержать **импорт по URL Google Drive** прямо в форме добавления.

_"Нажимает добавить и он пошёл закачал сюда из Google Drive."_ — Alexei

**Статус:** Принято как хорошая идея. Реализация — V1.0 или позже.

### R6. Сжатие — не нужно (пока)
Максим рассказал про опыт сжатия в кросс-промо (ограничение SDK 25 МБ). Для UA-закупки это неактуально: сети сами пережимают контент.

**Решение:** Сжатие не делаем. Возможно, добавим опциональную галочку позже.

_"Файлы могут быть любого размера."_ — Alexei

### R7. Архивация (sunset) старых креативов
Неактивные креативы (все кампании отключены давно) — переносить в дешёвое хранилище. Не удалять, чтобы у дизайнеров оставались референсы.

_"Я бы вообще ничего не удалял, чтобы у дизайнеров всегда были референсы."_ — Alexei

**Статус:** Принято концептуально. Детали реализации — позже.

### 3.2. Опыт кросс-промо (Максим Телига)
Максим уже строил похожую систему для кросс-промо:
- Хранение креативов на CDN
- Сжатие видео для SDK (ограничение 25 МБ)
- Версионность (до 50 версий у некоторых приложений)
- Проблема дубликатов iOS/Android
- Проблема нейминга на Google Drive

**Ключевые боли из его опыта:**
1. **Дубликаты** — одинаковые креативы для разных маркетов (iOS, Android, Amazon)
2. **Версионность** — до 50 версий (языки, праздники)
3. **Размер файлов** — видео до 200 МБ, иногда 4K
4. **Нейминг** — на Google Drive непонятные имена, не матчатся между платформами

### 3.3. Short name приложения
Текущее поле `short_name` (например, "CarCrash") — это не bundle ID и не название из App Store. Это внутреннее сокращение, которое выбирается произвольно.

_"Это не бандл. Это не название приложения в AppStore. Это из нашего внутреннего..."_ — Kapatsyn

**Вопрос:** Нужно ли формализовать правила формирования short_name?

### 3.4. Кросс-платформенность креативов
В 90% случаев один видеофайл подходит для всех сетей (стандартизированные разрешения). Но бывают исключения:
- iOS и Android могут иметь разный UI в геймплейных креативах
- Некоторые сети требуют специфических разрешений

## 5. Влияние на архитектуру

### 5.1. Потенциальные изменения модели данных

**Текущая модель:**
```
Game (code_1c) → Creative (concept) → Rendition (файл)
```

**Предложенная (R3):**
```
Game (code_1c) → Creative (concept) → CreativeRendition (связь)
                                            ↓
                                      MediaFile (независимый файл)
```

Медиафайл становится независимой сущностью, на которую ссылаются один или несколько креативов из разных игр/платформ.

**Риск:** Усложнение модели. Требует валидации: насколько реально переиспользование одного файла между iOS/Android.

## 6. Риски

1. **Переусложнение модели** — отвязка медиафайла от креатива добавляет гибкость, но может быть premature optimization. В реальности 90%+ креативов уникальны для одной игры.

2. **Нейминг ещё не финализирован** — появились новые идеи (хэш, код проекта), которые могут конфликтовать с уже согласованной конвенцией. Требуется финальное решение до начала разработки.

3. **Согласование с UA-менеджером** — Alexei ещё не поговорил с Vitalii Tereshchenko. Его фидбэк может повлиять на нейминг и процесс.
